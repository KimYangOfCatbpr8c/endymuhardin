表示

扩展 CollectionView 类支持从 OData 数据源加载和保存数据。

您可以使用 ODataCollectionView 类从 OData 服务加载数据，并将其用做任何 Wijmo 控件的数据源。

为了额外的获取完整的 CRUD 支持，您需要获取全部的 CollectionView 的功能， 包括排序，筛选，分页，以及分组。排序，筛选，分页功能既可以在服务器端完成也可以在客户端完成。

以下代码展示了如何实例化一个 ODataCollectionView，从数据源选择一些字段并在客户端提供排序。 请注意这里是如何通过使用‘Options’参数传递初始化数据， 这一过程和初始化控件相同：

var url = 'http://services.odata.org/Northwind/Northwind.svc';
var categories = new wijmo.odata.ODataCollectionView(url, 'Categories', {
  fields: ['CategoryID', 'CategoryName', 'Description'],
  sortOnServer: false
});

dataTypes any

获取或设置当加载数据时，一个用来强制数据类型的 JavaScript 对象。

表示字段名称和值的对象键值是 DataType 类型的值，该值表示如何将数据进行强制转换。

例如，以下代码创建了一个 ODataCollectionView 并指定 在数据库中存储为字符串类型的 ‘Freight’ 字段的值，应当被转换为数值类型； 另外三个日期类型的字段应当被转换为日期：

var orders = new wijmo.data.ODataCollectionView(url, 'Orders', {
  dataTypes: {
    Freight: wijmo.DataType.Number 
    OrderDate: wijmo.DataType.Date,
    RequiredDate: wijmo.DataType.Date,
    ShippedDate: wijmo.DataType.Date,
  }
});

该属性当数据库包含了和通常使用不一致的数据格式时，非常有用。

在大多数场景下，您不需要提供关于数据类型的信息， 因为 inferDataTypes 属性自动地处理了对 Date 类型值的转换。

如果您确实提供了显示的类型信息，则 inferDataTypes 将不会被应用上。 正因为如此，任何提供的数据类型信息应当是完整的，包括全部的具有 Date 类型的字段信息。

filterDefinition string

获取或设置一个字符串，指定包含在服务器端筛选数据的 OData 筛选器。

筛选器的定义语法在 OData 文档 中有描述。

例如，以下代码将使得服务器端返回 ‘CompanyName’ 字段以 ‘A’ 开头 ‘S’ 结尾的记录：

view.filterDefinition = "startswith(CompanyName, 'A') and endswith(CompanyName, 'B')";

筛选器定义可以被自动生成。 例如，FlexGridFilter 组件将检测其数据源是一个 ODataCollectionView 并且将自动地更新 filter 以及 filterDefinition 属性。

请注意即使是 filterOnServer 属性设置为 false，filterDefinition 也将生效。 这允许您有机会同时在服务器端和客户端针对同一集合应用筛选，这在某些场景下将非常有用。

例如，以下代码使用 filterDefinition 属性在服务器端进行筛选，同时通过 filter 属性在客户端对数据进行进一步的筛选。 集合将显示具有以字母‘C’开头，且单价大于 20 的记录。

var url = 'http://services.odata.org/V4/Northwind/Northwind.svc/';
var data = new wijmo.odata.ODataCollectionView(url, 'Products', {
  oDataVersion: 4,
  filterDefinition: 'startswith(ProductName, \'C\')', // 服务器端筛选
  filterOnServer: false, // 客户端筛选
  filter: function(product) { 
    return product.UnitPrice > 20;
  },
});
filterOnServer boolean

获取或设置一个值，该值决定是否筛选操作应当在服务器端完成还是在客户端完成。

使用 filter 属性在客户端执行筛选，使用 filterDefinition 属性在服务器端执行筛选。

在某些场景下，有可能需要应用独立的筛选器到客户端 以及 服务器端。

您可以完成以下步骤以达到此效果 （一）设置 filterOnServer 属性为 false 同时设置 filter 属性为一个筛选器函数（以启用客户端筛选），同时 （二）设置 filterDefinition 属性为一个筛选字符串（以启用服务器端筛选）。

oDataVersion number

获取或设置服务器使用的 OData 版本。

当前共有四个版本的 OData 服务，从版本 1.0 到版本 4.0。 版本 4.0 被大多数最新的服务所采纳，但是仍旧有大量的旧版服务仍然在正常运营。

如果您清楚您的服务实现，可以在创建 ODataCollectionView 时，设置 oDataVersion 属性为一个合适的值（从 1 到 4）（参见以下示例）。

var url = 'http://services.odata.org/Northwind/Northwind.svc';
var categories = new wijmo.odata.ODataCollectionView(url, 'Categories', {
  oDataVersion: 1.0, // 旧版的 OData 源
  fields: ['CategoryID', 'CategoryName', 'Description'],
  sortOnServer: false
});

如果您不清楚您所用到的服务所采用的 OData 实现的版本（也许 您正在做一个 OData 资源管理器程序），那么请不用指定此版本号。 在这种情况下，ODataCollectionView 将从服务器获取该信息。 这一操作将需要额外的信息请求，但是每一个服务 URL 仅请求一次， 因此所带来的影响甚微。

sortComparer Function

获取或设置一个函数用于在排序的基准。 如果提供，这个函数应该接受两个任意类型的值。 返回-1, 0, 或者 +1，制定第一个值小于，等于，或大于第二个值。 如果返回null，标准内置的比较器会被使用。 sortComparer 属性允许你使用自定义排序算法，以保证在某些情况下，用于的需求不仅仅是按照字符串排序。比如，

Dave Koele's Alphanum algorithm 它将字符串拆分为字符串串和数字。然后数字按照值排序，字符串按照ASCII码排序。Dave 把这个叫做“自然顺序”。 这个例子展示了 sortComparer 属性的常见用法。 

// 使用自定义排序创建一个 CollectionView。
var dataCustomSort = new wijmo.collections.CollectionView(data, {
  sortComparer: function (a, b) {
    return wijmo.isString(a) && wijmo.isString(b)
      ? alphanum(a, b) // custom comparer used for strings
      : null; // use default comparer used for everything else
  }
});


(继承自 CollectionView).
sortConverter Function

获取或设置排序时用作转换数据的函数。

如果提供了，函数应该被当作SortDescription ，数据项和转换值的参数，并且应该返回转换的值。

此属性提供了一种自定义排序的方法。例如，FlexGrid 控件使用该属性来排序映射列的显示值，而不是由原始值。

例如，下面的代码引起CollectionView 来排序'country'属性，该属性包含国家代码整数 ，使用相应的国家代码名称：

var countries = 'US,Germany,UK,Japan,Italy,Greece'.split(',');
collectionView.sortConverter = function (sd, item, value) {
  if (sd.property == 'countryMapped') {
    value = countries[value]; // 转换国家id为名称
  }
  return value;
}

trackChanges boolean

获取或设置一个值，该值确定该控件是否对数据进行跟踪更改。

如果trackChanges 被设置为真，CollectionView 对数据的更改进行跟踪，并且通过 itemsAdded， itemsRemoved，和itemsEdited 集合暴漏它们。

跟踪变化在多种情况下是非常有用的，当用户确认该修改是有效的之后你需要更新服务器。

在提交或取消更改后，使用clearChanges 方法来清除itemsAdded， itemsRemoved 和itemsEdited 集合。

CollectionView 只跟踪正式的CollectionView 方法被使用时的改变（editItem/commitEdit，addNew/commitNew 和remove）。 直接对数据的更改不被跟踪。

useStableSort boolean

获取或设置是否使用稳定排序算法。

稳定的排序算法以相等的键值保持记录的相对顺序。 例如，考虑一个“Amount”字段集合的对象。. 如果你按"Amount"对该集合进行排序，,一个稳定的排序将保留原始的记录的顺序和相同Amount值.

该属性默认为假，使类使用JavaScript内置的排序方法，非常快，但不稳定。设置 stableSort 为真，从百分之三十到五十增加排序次数，这些对大的集合有意义。

addNew​(): any

创建一个新项目并将其添加到集合中。

该方法没有参数。它创建了一个新的项目，将它添加到集合中，并防止刷新操作，直到新项目使用commitNew 方法提交或者使用cancelNew 方法取消。

下面的代码显示addNew 方法被典型的使用:

// 创建新项目，将其添加到集合中
var newItem = view.addNew();
// initialize the new item
newItem.id = getFreshId();
newItem.name = 'New Customer';
// 提交新的项目，这样可以刷新视图 
view.commitNew();

您还可以通过将它们推到sourceCollection 中，然后调用refresh 方法来添加新项目。addNew的主要优点是在用户交互的情况下 （像在数据网格中添加新的项目），因为它给用户取消添加操作的能力。它还可以防止新的项目被排序或过滤掉，直到添加操作提交。

(继承自 CollectionView).
currentChanged 在当前项目改变后发生。
(继承自 CollectionView).
currentChanging CancelEventArgs 在当前项目改变之前发生。
(继承自 CollectionView).
error RequestErrorEventArgs 当在读取或写入数据是发生一个错误时触发。
loaded EventArgs 当 ODataCollectionView 结束加载数据时触发。
loading EventArgs 当 ODataCollectionView 开始加载数据时触发。
pageChanged 在页面索引改变之后发生。
(继承自 CollectionView).
pageChanging PageChangingEventArgs 在页面索引改变之前发生。
(继承自 CollectionView).
sourceCollectionChanged EventArgs 在 sourceCollection 属性改变后触发。
(继承自 CollectionView).
sourceCollectionChanging CancelEventArgs 在 sourceCollection 属性更改前触发。
(继承自 CollectionView).
